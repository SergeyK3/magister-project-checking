# Инструкция: проект в Google Cloud Console для Magister Project Checking

Документ описывает создание облачного проекта, включение API и настройку OAuth **для десктопного Python-приложения** (установленное приложение, поток «Installed App»), как в пилотном этапе с личным аккаунтом Google.

**Полезные ссылки**

- [Google Cloud Console](https://console.cloud.google.com/)
- [OAuth 2.0 для установленных приложений](https://developers.google.com/identity/protocols/oauth2/native-app)

---

## 1. Вход и создание проекта

1. Откройте [Google Cloud Console](https://console.cloud.google.com/) и войдите в Google-аккаунт (для пилота логично использовать тот же аккаунт, с которым будете тестировать приложение, например `uz1.nursultan@gmail.com`).
2. В верхней панели нажмите на **селектор проекта** (название текущего проекта или «Select a project»).
3. Нажмите **New Project** / **Новый проект**.
4. Укажите **Project name** (например, `Magister Checking Dev`). Поля **Organization** и **Location** можно оставить по умолчанию, если не требуется иное.
5. Нажмите **Create** / **Создать**. Через несколько секунд проект появится в списке — снова откройте селектор проектов и **выберите созданный проект**, чтобы все дальнейшие действия выполнялись в нём.

---

## 2. Включение API

Нужны как минимум **Google Drive API** (чтение метаданных) и **Google Docs API** (чтение и запись документов с таблицами). **Google Sheets API** не обязателен, если вы не используете устаревший сценарий записи в Таблицы.

Для каждого нужного API:

1. В верхней строке поиска введите название и откройте карточку **внутри текущего проекта**:
   - **Google Drive API**
   - **Google Docs API**
2. На странице API нажмите **Enable** / **Включить**.

Если кнопка называется **Manage** / **Управлять**, API уже включён — переходите к следующему.

---

## 3. Экран согласия OAuth (OAuth consent screen)

1. Откройте меню ☰ → **APIs & Services** → **OAuth consent screen** (или по-русски: **API и сервисы** → **Экран согласия OAuth**).
2. Выберите тип пользователя:
   - **External** — подходит для обычного Gmail и внутреннего тестирования. Нажмите **Create**.
3. Шаг **App information**:
   - **App name** — отображаемое имя (например, `Magister Report Checker`).
   - **User support email** — ваш адрес почты.
   - **App logo** — по желанию, не обязательно для разработки.
   - Внизу: **Developer contact information** — email для связи с разработчиком (обычно тот же).
4. Шаг **Audience** / аудитория:
   - Если приложение в режиме **Testing**, в разделе **Test users** нажмите **Add users** и добавьте **все Google-аккаунты**, под которыми будете открывать окно входа OAuth (обязательно тот, с которым тестируете, например `uz1.nursultan@gmail.com`).  
   Пока приложение не опубликовано и не прошло проверку Google, **другие** пользователи не смогут пройти авторизацию, если их нет в списке тестовых.
5. Шаг **Scopes**:
   - На первом этапе можно **не добавлять** scope вручную на экране согласия: клиент запросит нужные области при первом запуске из кода. При желании позже добавьте read-only для Drive/Docs и при необходимости Sheets — так пользователю будет понятнее, что запрашивает приложение.
6. Завершите мастер (**Save and Continue** до конца).

**Важно:** для приложений в статусе **Testing** действуют ограничения Google на срок жизни refresh token и доступ. Для долгой автономной работы без повторного входа позже может понадобиться смена режима или согласование с политикой Google; на этапе отладки достаточно тестового пользователя и локального сохранения токена.

---

## 4. Учётные данные OAuth: тип «Desktop»

1. Перейдите в **APIs & Services** → **Credentials** (**Учётные данные**).
2. Нажмите **+ Create Credentials** → **OAuth client ID**.
3. Если консоль сообщит, что нужно сначала настроить экран согласия — вернитесь к разделу 3 и завершите настройку.
4. **Application type** выберите **Desktop app** (в русской локали может быть «Настольное приложение»).
5. **Name** — произвольное (например, `Desktop client`).
6. Нажмите **Create**. В диалоге можно сразу нажать **Download JSON** и сохранить файл.

Сохраните JSON **вне репозитория** или в каталоге, указанном в `.gitignore` (например, `credentials/client_secret.json`). **Не публикуйте** этот файл в Git и не отправляйте в открытые чаты.

## 5. Связь с кодом Python (кратко)

- Используется поток **Installed application**: при первом запуске откроется браузер, вы выберете аккаунт и разрешите доступ; библиотека `google-auth-oauthlib` сохранит токен локально (часто `token.json` рядом с секретом или в отдельном файле — как настроите в коде).
- Актуальный набор в коде проекта (см. `magister_checking/auth.py`):
  - `https://www.googleapis.com/auth/drive.readonly` — просмотр файлов на Диске;
  - `https://www.googleapis.com/auth/documents` — **чтение и запись** Google Docs (в т.ч. заполнение таблиц в документе).  
  Раньше использовались `documents.readonly` и Sheets — после смены ТЗ нужен **полный** доступ к Docs и повторный вход (`login`), иначе `token.json` будет с недостаточными scope.

## 6. Типичные ошибки и что проверить

| Симптом | Что сделать |
|--------|-------------|
| Доступ заблокирован, приложение не проверено | Используйте аккаунт из **Test users**; либо уменьшите/уточните scope; для публичного доступа нужна проверка приложения в Google. |
| `redirect_uri_mismatch` | Для Desktop-клиента URI задаётся библиотекой (часто `http://localhost` с портом). Убедитесь, что в коде используется тип **Desktop**, а не Web с неверным redirect. |
| 403 / API not enabled | В селекторе проектов выбран **тот же** проект, где включены Drive, Docs и Sheets. |
| Другой пользователь не может войти | Добавьте его в **Test users**, пока приложение в режиме Testing. |
| Нет квот | Для перечисленных API обычно хватает бесплатного уровня; привязка платёжного аккаунта часто не требуется для разработки. |

---

## 7. Безопасность

- Не коммитьте JSON клиента и файлы с токенами (`token.json`, `credentials.json` с секретами).
- При утечке клиентского секрета создайте в консоли **новый** OAuth client ID и отзовите/удалите скомпрометированный.
- Для продакшена рассмотрите отдельный проект Cloud, отдельную учётную запись или согласование с ИТ вуза.
