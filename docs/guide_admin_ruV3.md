# Руководство администратора. V3

Проект «Проверка магистерских проектов»

Дата: 06.05.2026

---

# 1. О проекте

Для автоматизации отдельных этапов аттестации магистрантов используется Telegram-бот `@magistrcheckbot`.

Описание проекта: https://docs.google.com/document/d/1Dq0MJL1hrdNG4XYo-BNKMrmRfnO3-ndeTo42vg1GjUM/edit?usp=sharing

Система предназначена для:

* регистрации участников проекта;
* проверки промежуточных отчетов;
* проверки ссылок на материалы магистерского проекта;
* анализа диссертаций;
* формирования справок и отчетов;
* дистанционного контроля со стороны научных руководителей и администраторов.

---

# 2. Архитектура системы

Система состоит из:

* Telegram-бота;
* ядра обработки;
* Google Таблицы;
* Google Docs API;
* Google Drive API.

Google Таблица является основным источником данных (source of truth).

---

# 3. Основные роли

## Магистрант

Может:

* зарегистрироваться;
* запускать проверку проекта;
* получать справку о состоянии проекта;
* отправлять персональное сообщение администраторам через бота.

---

## Научный руководитель

Может:

* просматривать состояние проектов своих магистрантов;
* получать списки зарегистрированных и незарегистрированных подопечных;
* запускать проверку по подопечному командой `/status` с ФИО (лист «Магистранты»).

Указать номер строки или чужое ФИО прямо в `/spravka` может только администратор; научруку для чужой строки нужен `/status ФИО`, а не `/spravka Фамилия`.

---

## Администратор

Может:

* просматривать любые проекты;
* отправлять замечания магистрантам;
* формировать PDF-карточки;
* выполнять ручной контроль.

---

# 4. Основной пользовательский сценарий

## Шаг 1. Идентификация

Пользователь запускает:

```text
/start
```

Бот:

1. проверяет Telegram ID;
2. определяет роль;
3. при необходимости запрашивает ФИО;
4. выполняет поиск по Google Таблице;
5. привязывает Telegram ID;
6. показывает меню.

---

## Шаг 2. Проверка проекта

Команда:

```text
/spravka
```

запускает:

* проверку промежуточного отчета;
* проверку ссылок;
* анализ диссертации;
* формирование отчета.

Повторный запуск `/spravka` выполняет повторную проверку проекта.

---

# 5. Ролевое меню

Основной сценарий для всех ролей один:

```text
/start → бот определяет роль → показывает меню этой роли кнопками
```

Ручные slash-команды сохраняются как техническая совместимость и описаны в `docs/commands.md`, но для обычной работы сотрудник использует кнопки меню.

---

## Общие кнопки

| Кнопка | Назначение |
| ------ | ---------- |
| `Справка / проверка` | запускает проверку или меню форматов с учётом роли |
| `Проверить статус` | магистранту показывает статус своей строки; администратору запрашивает строку или ФИО из листа `Регистрация` |
| `Продолжить регистрацию` | продолжает или обновляет анкету магистранта |
| `Написать администратору` | магистрант отправляет сообщение всем активным администраторам |
| `Помощь` | показывает подсказку по роли |

---

## Меню научного руководителя

| Кнопка | Назначение |
| ------ | ---------- |
| `Проверить магистранта` | проверка/статус подопечного |
| `Кто не зарегистрировался` | список незарегистрировавшихся подопечных |
| `Кто зарегистрировался` | список зарегистрировавшихся подопечных |
| `Справка / проверка` | проверка своей строки, если научрук также есть в «Регистрация» |
| `О проекте` | ссылка на описание проекта |

---

## Меню администратора

| Кнопка | Назначение |
| ------ | ---------- |
| `Справка / проверка` | меню форматов справки и проверка строк с правами администратора |
| `Сообщение магистранту` | напоминание одному магистранту |
| `Групповое напоминание` | сообщение нескольким строкам выбранного листа |
| `Сообщение научруку` | стандартное сообщение `нет регистрации` или свободное сообщение научруку |
| `Сформировать карточку проекта` | PDF-карточка по строке или ФИО |
| `Сводка` | административная сводка |
| `О проекте` | ссылка на описание проекта |
| `Помощь` | подсказка по роли |

Команды `/список` и `/ошибка` в текущей версии бота **не реализованы**; массовая отправка замечаний по номерам строк из старых сценариев заменена кнопками `Групповое напоминание` и `Сообщение магистранту`.

После нажатия `Групповое напоминание` бот последовательно спрашивает:

1. имя листа: `Регистрация` или `Магистранты`;
2. номера строк выбранного листа;
3. текст сообщения. В тексте можно использовать `{fio}` для подстановки ФИО получателя.

После нажатия `Сообщение научруку` бот сначала просит выбрать научрука, затем предлагает тип сообщения: `нет регистрации` или `свободное сообщение`.

---

# 6. Сообщение магистранту

После нажатия `Сообщение магистранту`:

1. администратор указывает цель (строка или ФИО);
2. выбирает шаблон напоминания или вводит свой текст;
3. бот показывает предпросмотр;
4. доступны кнопки **Отправить** и **Отмена**;
5. сообщение уходит магистранту в личку; в тексте напоминания обычно есть призыв снова выполнить `/справка` после правок.

---

# 6A. Сообщение администратору от магистранта

Сценарий запускается кнопкой `Написать администратору` в меню магистранта или командой `/admin_message`.

Бот последовательно делает следующее:

1. проверяет, что отправитель привязан к строке листа `Регистрация`;
2. просит прислать текст сообщения одним сообщением;
3. показывает предпросмотр;
4. после подтверждения отправляет сообщение всем активным администраторам из листа `Администраторы`.

Администратор получает не только текст, но и служебный контекст отправителя:

* ФИО;
* номер строки `Регистрация`;
* группу;
* `telegram_id`;
* `username`.

Если в листе `Администраторы` нет активных корректных `telegram_id`, бот сообщает магистранту, что доставка невозможна.

# 7. Групповое напоминание

Сценарий запускается кнопкой `Групповое напоминание` в панели `/admin` или командой `/student_message_bulk`.

Бот последовательно спрашивает:

1. имя листа: `Регистрация` или `Магистранты`;
2. номера строк выбранного листа, начиная со строки 2;
3. текст сообщения для рассылки.

Номера строк можно вводить через пробел, запятую или с новой строки, например:

```text
5 7 9 10
```

или:

```text
12,15,18
```

В тексте сообщения можно использовать `{fio}` — бот заменит этот фрагмент на ФИО получателя из выбранной строки. Пример:

```text
Здравствуйте, {fio}! Проверьте, пожалуйста, справку и внесите исправления.
```

После ввода текста бот показывает предпросмотр и список получателей. Для отправки нужно нажать **Отправить всем**; для отмены — **Отмена** или `/cancel`.

Требования к выбранному листу:

* в строке должен быть заполнен `telegram_id`;
* для подстановки `{fio}` должна быть заполнена колонка ФИО;
* лист `Магистранты` должен быть настроен в конфигурации как `MAGISTRANTS_WORKSHEET_NAME`.

Для листа `Регистрация` бот, как и в одиночном напоминании, может приложить HTML-снимок последней проверки, если он найден на Drive. Для листа `Магистранты` HTML-снимок не прикладывается, потому что номер строки относится к другому листу.

# 8. Сообщение научруку

Сценарий запускается кнопкой `Сообщение научруку` в панели `/admin` или командой `/supervisor_message`.

Бот последовательно делает следующее:

1. просит ввести ФИО научрука или номер строки листа `научрук`;
2. находит строку научрука; `telegram_id` для формирования шаблона не обязателен;
3. предлагает выбрать тип сообщения:
   * **нет регистрации** — стандартное сообщение со списком магистрантов без регистрации;
   * **свободное сообщение** — текст, который администратор вводит вручную;
4. показывает администратору предпросмотр;
5. если `telegram_id` заполнен, после нажатия **Отправить научруку** бот отправляет сообщение научруку в личку Telegram;
6. если `telegram_id` пустой или некорректный, бот всё равно показывает шаблон и сообщает, что его нужно переслать через другой канал связи.

Тип **нет регистрации** фильтрует лист `Магистранты` по выбранному научруку и колонке `Регистрация` со значением `нет`.

В список попадают только строки листа `Магистранты`, где:

* научный руководитель совпадает с выбранным научруком;
* в колонке `Регистрация` указано `нет`.

Для каждого магистранта в стандартном сообщении показываются ФИО и телефон.

Тип **свободное сообщение** не использует список магистрантов: бот просит прислать полный текст одним сообщением, показывает предпросмотр и после подтверждения отправляет этот текст выбранному научруку.

Если ФИО научрука найдено в нескольких строках листа `научрук`, бот попросит указать номер строки. Если `telegram_id` научрука пустой или некорректный, Telegram-отправка не выполняется, но шаблон сообщения всё равно формируется для пересылки через email, WhatsApp или другой канал.

# 9. Этапы проверки

## Этап 1. Проверка промежуточного отчета

Проверяется:

* доступность ссылки;
* тип ссылки;
* наличие фразы «Промежуточный отчет».

---

## Этап 2. Проверка ссылок проекта

Проверяются:

* папка проекта;
* заключение ЛКК;
* диссертация;
* публикация.

---

## Этап 3. Анализ диссертации

Определяются:

* число страниц;
* число источников;
* соответствие оформлению.

---

## Этап 4. Извлечение данных диссертации

Определяются:

* название диссертации;
* язык диссертации.

---

# 10. Ручной ввод ссылок

Если Google Docs API не смог извлечь ссылки автоматически, администратор может вручную внести URL в лист «Регистрация».

После сохранения администратор повторно запускает проверку, например:

```text
/spravka 25
```

или `/spravka Иванов И.И.` / `/recheck …` — см. `docs/commands.md`.

---

# 11. Нормализация групп

На вкладке «Магистранты» используется поле:

```text
Группа норм
```

Администратор вручную указывает стандартизированное название группы.

---

# 12. Snapshot Architecture

Система может сохранять JSON-снимки проекта.

Snapshot содержит:

* метаданные;
* результаты проверок;
* ссылки;
* статусы этапов.

Snapshot может использоваться:

* для повторного формирования отчетов;
* для PDF-карточек;
* для восстановления состояния проверки.

---

# 13. CLI-only операции

Тяжёлые массовые операции, скрипты обслуживания и правки данных по умолчанию выполняются через CLI и скрипты репозитория.

Исключение: у администратора в боте есть узкий набор команд синхронизации таблиц (`/sync_dashboard`, `/sync_magistrants`) и статистики — см. `docs/commands.md`.

---

# 14. Устаревшие названия в документации

В старых черновиках встречались команды `/список`, `/ошибка`, обещание «удалённого» `/recheck`. В **текущем** `@magistrcheckbot`:

* `/recheck` и `/cancel` **есть**;
* повторная проверка также доступна через `/spravka` и кнопку «перепроверить» под отчётом (если бот её показал).

Канонический перечень команд — `docs/commands.md`; при расхождении с этим руководством приоритет у `commands.md`.
