# Handoff — ТЗ MagistrCheck [`doptz20260430.md`](doptz20260430.md): что уже есть и что не совпадает с черновиком

**Дата:** 2026-04-30  
**Источник требований:** [`docs/doptz20260430.md`](doptz20260430.md) (черновик от 30.04.2026).

## 0. Статус против старого handoff

В [`handoff_2026-04-30_magistrants_sync_supervisor_commands.md`](handoff_2026-04-30_magistrants_sync_supervisor_commands.md), §4, было написано, что PIN по ТЗ **не внедрён**. Это **устарело**: одноразовый PIN (**§5–11 MVP** черновика) реализован в процессе бота (память, SHA-256, лог). Ниже — полная решётка по разделам `doptz20260430.md`.

---

## 1. Сопоставление с ТЗ по разделам

| Раздел ТЗ | Суть требования | В продукте сейчас |
|-----------|----------------|-------------------|
| **§1 Назначение** | Идентификация Telegram, привязка к Sheets, роли, статусы | **Да.** Роли: листы «Администраторы» / **`научрук`**, магистрант по «Регистрации» + мастер-лист «Магистранты» при `MAGISTRANTS_WORKSHEET_NAME`. |
| **§2 Источник данных** | Конкретный `SPREADSHEET_ID` и набор листов | Частично моделью: книга задаётся **`SPREADSHEET_ID`** в `.env`; названия листов — **`WORKSHEET_NAME`** (обычно «Регистрация»), **`научрук`**, **`Администраторы`**, опционально лист **`MAGISTRANTS_WORKSHEET_NAME**`. Совпадение с ID из черновика **не контролируется** кодом. |
| **§3 Логика /start** | Не найден tg_id → запуск регистрации (FSM) | **Частично.** У неизвестного пользователя после **`/start`** — **выбор роли** (магистрант / админ / научрук) и разные сценарии (**привязка по ФИО**, регистрация с нуля, claim для админа/научрука), а не единый «поток неизвестный → только анкета», как на схеме §12. |
| **§4 FSM (имена состояний)** | Перечень `STATE_*` до `STATE_BLOCKED` | **Не реализовано как именованный автомат.** В коде состояния `ConversationHandler` — `ROLE_PICK`, `BIND_*`, `ASK_FIELD`, **`PIN_VERIFY_INPUT`**, и др.; нет побитового совпадения с перечнем ТЗ и нет явных состояний `STATE_BLOCKED` / `STATE_PIN_ERROR` как отдельных узлов. |
| **§5 PIN + таблица `verification_codes`** | 4–6 цифр, 5 мин, 3 попытки, hash; поля phone, pin_hash, expires_at, attempts, tg_id | **PIN:** параметры MVP совпадают (**`magister_checking/bot/pin_verify.py`**). **`verification_codes`** как **устойчивое хранилище** (БД / таблица) **нет** — только **in-memory** на процесс (**§11 MVP** это допускает). |
| **§6 Запись tg_id** | После успеха: магистрант → **лист Магистранты**; научрук → **Научруки** | **Расходится с черновиком.** Фактическая запись после PIN: · **магистрант**: **`telegram_id` в строку листа «Регистрация»** (и логика анкеты там же), а не первичная запись на мастер-лист «Магистранты»; синк **`sync_magistrants_registration_status`** обновляет по мастеру **телефон + колонку «Регистрация»**, а не задаёт канонический сценарий «все tg_id только на листе Магистранты». · **научрук/админ** после PIN → **`telegram_id`** в строках листов **`научрук`** или **«Администраторы»** через `attach_telegram_to_row` (название листа научников в коде — **`научрук`**). Если ТЗ считало обязательной колонку **tg_id на «Магистрантах»** при каждой регистрации — её **нет в текущей реализации** как целевая точка сохранения. |
| **§7 Меню** | Интерактивные блоки меню для трёх ролей | **Частично.** Ответ интерфейса — **`set_my_commands` / текст `/help`** и шаги диалогов; для **магистранта** есть клавиатура админки (не для научрука). Отдельного **Reply-клавиатурного «меню как в ТЗ»** (три блока один в один с подпунктами) **нет.** Содержание в целом перекрывается командами: научрук — **`/unreg`**, **`/reg_list`**, **`/status`**; админ — много команд + **`/stats`** вместо формулировки «просмотр пользователей»; магистрант — **`/register`**, **`/start`** (продолжение анкеты), **`/status`**, **`/recheck`** — но нет явной команды **«обновить статус»** как отдельного пункта меню. |
| **§8 Логирование** | Авторизация, регистрация, PIN, ошибки; пример JSON | **Частично.** События **`pin_issued`** / **`pin_verify`** пишутся структурированно (**см. `guide_admin_ru.md` §8.6**). Полный единый аудит «авторизация/регистрация» в одном формате по всем шагам **не сведён** к одному контракту полей; ошибки — через стандартный лог + при необходимости **`BOT_ALERT_CHAT_IDS`**. |
| **§9 Ошибки** | PIN истёк; пользователь не найден; попытки | **Частично.** Пользователь получает понятные ответы; в лог **`pin_verify`** уходит **`expired`** / **`locked`** / **`wrong`**, не обязательно строка **`failed`** как в примере JSON черновика. |
| **§10 Ограничения** | 1 tg_id = один пользователь; строгий ФИО | **В целом да** для привязок и анкеты; узкие кейсы множественных совпадений ФИО обрабатываются сообщениями «уточните / обратитесь к админу». |
| **§11 MVP** | PIN в лог; память допустима | **Выполнено.** Открытый PIN в логе при **`pin_issued`**; данные попытки в RAM. |
| **§12 Поток** | `/start` → tg → меню или регистрация → **ФИО** → PIN → запись tg_id → меню | **Частично.** PIN включается **после «да»** при привязке к строке и **перед сохранением анкеты** по сводке, а не только в одном жёстком порядке «сначала ФИО потом только PIN». Порядок согласован с текущим UX продукта, но **не идентичен** тексту §12. |

## 2. Короткий список «осталось / расходится» (для следующей итерации)

Приоритет трактует заказчик; ниже — нейтральное перечисление зазоров.

1. **Модель листов из §2 §6**: явно решить — нужно ли записывать **`telegram_id` на лист «Магистранты`** (или другое поле статуса) по ТЗ, или официально зафиксировать книгу регистрации как **единственную** точку truth для `telegram_id` магистранта.
2. **FSM по именам §4** — либо привести документацию к фактическим состояниям PTB, либо рефакторинг под явный слой состояний (если требуется аттестация/аудит по схеме ТЗ).
3. **Персистентное хранилище PIN** (таблица **`verification_codes`** или Sheets) — вне MVP; для SMS и кластера из нескольких инстансов бота RAM **недостаточен**.
4. **Меню §7** — при необходимости: клавиатуры под роли, отдельная команда «обновить статус» для магистранта; «просмотр пользователей» для админа (сейчас ближе всего **`/stats`** + работа в таблице).
5. **Единый контракт логов §8** для событий входа/регистрации (как для PIN).
6. **Поток §12** — либо обновить ТЗ под фактический порядок (роль → привязка/анкета → PIN в двух местах), либо унифицировать сценарий в коде.
