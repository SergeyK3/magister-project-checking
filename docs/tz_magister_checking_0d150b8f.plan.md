# Техническое задание: автоматизация проверки сведений о магистерском проекте

## 0. Пилотная среда и эталонные материалы (этап 1)

**Учётная запись Google для отработки процессов:** `uz1.nursultan@gmail.com` (личный аккаунт разработчика на первом этапе).

**Тестовая папка на Google Диске** (общий доступ по ссылке): [тестовая папка1](https://drive.google.com/drive/folders/1crKcql5hmz0VYvqgRY8_sTaFeIguw836?usp=sharing).

**Содержимое папки (по состоянию на подготовку ТЗ):** эталонные Google-шаблоны **Приложений 1–3** для парсинга и маппинга колонок; промежуточные отчёты по фамилиям (Ахметжанов, Сапабаева, Серикова, Шахметов); документ «Промпт для бота»; тестовая сводная таблица по магистерским проектам. Возможны дополнительные файлы — перед разработкой актуализировать список и зафиксировать id документов/таблиц в конфигурации проекта.

**Вывод для реализации:** аутентификация API на этапе 1 — **OAuth 2.0** от имени указанного аккаунта (или экспорт credentials в безопасное хранилище вне репозитория); доступ к файлам с режимом «всем по ссылке» подтверждён для тестовой папки.

## 1. Основание и цель

**Источник требований:** [Временное положение о порядке сбора информации](https://docs.google.com/document/d/1fkikkPc7qWAW9tj4UkVvcLwpf412QsaXTKPUW_sQhPQ/edit) (МУА, КОЗМ, промежуточная аттестация).

**Проблема:** на аттестации комиссия опирается на устные ответы магистранта; возможны неполные или искажённые сведения по обзору, ЛКБ, публикациям, этапам работы и календарному плану.

**Цель системы:** формализовать подачу материалов (структура Google Диска + «Промежуточный отчёт») и **автоматически собирать проверяемые факты** по ссылкам из отчёта, заполняя сводные данные для председателя комиссии (аналог **Приложения 3**), с минимизацией ручного ввода и субъективных оценок там, где возможна объективная проверка.

## 2. Заинтересованные стороны и сценарии

| Роль | Действия (по положению) | Роль в ПО |
|------|-------------------------|-----------|
| Магистрант | Папка «Магистерский проект», подпапки, документы, доступ «всем по ссылке», заполнение отчёта (Прил. 1) | Пользователь процесса; объект контроля |
| Завуч / комиссия | Получение ссылки на отчёт | Потребитель результата |
| Председатель / секретарь | Сводная таблица по проектам (Прил. 2), свод по этапам (Прил. 3) | Пользователь бота и таблицы |

**Основной сценарий ПО:** оператор указывает ссылку на Google Документ **«Промежуточный отчёт магистранта»** (и/или строку в **Прил. 2** с такой ссылкой) → бот читает доступные по ссылке документы → заполняет строку/строки в целевой **Google Таблице** (формат **Прил. 3** + при необходимости дополнительные служебные колонки).

**Массовый сценарий:** пакетная обработка списка ссылок из колонки «Промежуточный отчёт» сводной таблицы (Прил. 2).

## 3. Объект автоматизации и границы

**Входит в объём:**

- Разбор отчёта магистранта: извлечение гиперссылок и текстовых полей (ЛКБ да/нет, статья по обзору да/нет, куда/когда подана и т.д.).
- По ссылке на **диссертацию** (Google Doc): подсчёт объёма (страницы или эквивалент по API), **наличие заголовков разделов** согласно п. 5–6 положения (Введение; Материалы и методы; Обзор литературы; Результаты; Обсуждение и выводы; Заключение; Приложения).
- **Обзор литературы:** наличие раздела; оценка объёма в страницах/словах (уточнить в проекте правило границ раздела); **количество источников** — по парсингу списка литературы / раздела «Ссылки» (эвристики: нумерованный список, GOST-подобные строки; согласовать с эталонным шаблоном диссертации из тестовой папки).
- **Оформление:** проверка через Google Docs API стилей абзацев (шрифт Times New Roman, кегль 14, межстрочный интервал 1.0) — **выборочно по основному тексту** или по всему документу (уточнить, чтобы не ломаться на титульном листе).
- **Календарный план** (файл в «Сопутствующие документы»): извлечение таблицы; проверка, что в колонке сроков **нет диапазонов** (например «ноябрь–декабрь»), а **конкретные даты**; флаг отклонения (п. 14 положения).
- **Публикации:** если в отчёте есть ссылки на файлы в папке «Публикации» — учитывать метаданные (для неопубликованного — вторая строка после названия: издание, дата отправки) — по возможности из Google Doc / PDF текста.

**Ограничения и вне объёма (явно зафиксировать в ТЗ):**

- Бот **не заменяет** экспертную оценку содержания; только структура, метрики, ссылки, формальные признаки.
- Доступ только к тем файлам, на которые дана ссылка с правом **чтения** («всем по ссылке»). Если магистрант забыл открыть доступ — в таблице фиксируется ошибка доступа.
- PDF на Диске: извлечение текста зависит от типа PDF (текстовый vs скан); для сканов — либо исключение, либо отдельная задача (OCR).
- Подтверждение факта публикации на сайте журнала без парсинга конкретных издательств — по желанию заказчика отдельным этапом.

## 4. Функциональные требования

1. **Ввод:** URL документа «Промежуточный отчёт»; опционально — идентификатор строки в сводной таблице (ФИО, группа).
2. **Парсинг отчёта:** сопоставление блоков отчёта с полями Прил. 1 (ЛКБ, статья по обзору, диссертация, статья по результатам и т.д.) по структуре **эталонного шаблона** из тестовой папки.
3. **Обход ссылок:** по каждой релевантной ссылке — определение типа (Google Doc, Google Sheet, PDF, папка Drive) и вызов соответствующего обработчика.
4. **Диссертация:** вычисление метрик и чек-листа разделов; сохранение в выходную строку.
5. **Календарный план:** парсинг таблицы, валидация формата дат, предупреждения.
6. **Вывод:** запись в Google Sheets в колонки, соответствующие **Прил. 3** (и расширения: ссылки на проверенные документы, время проверки, статус ошибки), в соответствии с **эталонной таблицей** из тестовой папки.
7. **Логирование:** для каждого магистранта — что удалось прочитать, что не открылось, какие эвристики сработали.

## 5. Нефункциональные требования

- **Язык интерфейса оператора:** русский.
- **Безопасность:** секреты (OAuth token, client secrets) не в репозитории; минимальные права на Drive/Docs/Sheets.
- **Производительность:** обработка одного отчёта в разумные сроки с учётом квот Google API (пакеты с задержкой/повтором).
- **Воспроизводимость:** фиксированные версии зависимостей; возможность запуска из командной строки или простого UI (на усмотрение заказчика).

## 6. Интеграции и стек (рекомендация для реализации)

- **Google APIs:** Drive API (файлы по id из ссылок), Docs API (структура, стили, текст), Sheets API (запись свода).
- **Аутентификация (этап 1):** OAuth 2.0 для `uz1.nursultan@gmail.com`. Для промышленного варианта — отдельно согласовать сервисный аккаунт или учётную запись организации.
- **Реализация:** Python 3.x в репозитории `D:\MyActivity\MyInfoBusiness\MyPythonApps\12 MagisterProjectChecking`.

## 7. Структура данных выходной таблицы (минимум)

Соответствие **Прил. 3** (колонки уточнять по эталону в тестовой папке): ФИО; ЛКБ (из отчёта + при наличии ссылки на файл — факт открытия); статья (название, формат); диссертация — да/нет, страницы; число источников; наличие разделов «Обзор», «Результаты», «Обсуждение»; общее число страниц.

**Дополнительно (рекомендуется):** `checked_at`, `errors`, `calendar_plan_ok`, `calendar_plan_notes`.

## 8. Этапы разработки и приёмки

1. **Прототип:** отчёты и диссертации из тестовой папки — разделы, страницы, источники (ручная сверка с эталоном).
2. **Календарный план:** правила дат и отчёт о нарушениях.
3. **Пакетная обработка** + запись в шаблон Sheets.
4. **Документация для комиссии:** как запускать, что означают столбцы и ошибки.

**Критерии приёмки:** на контрольном наборе из N отчётов (N согласовать; в тестовой папке уже есть несколько примеров) автоматически заполненные поля совпадают с ручной проверкой не ниже порога по заранее определённым метрикам (например разделы и страницы — 100%, источники — согласованный допуск из-за эвристик).

## 9. Риски

- Разнородное оформление списка литературы → занижение/завышение счётчика.
- Нестандартные названия разделов → ложное «отсутствие раздела».
- Квоты и лимиты API Google.
- Поведение API при доступе «только по ссылке» — проверить прототипом на документах из тестовой папки под учётной записью этапа 1.
