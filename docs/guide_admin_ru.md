# Руководство администратора Telegram-бота проверки магистерских проектов

Документ для тех, кто **управляет процессом** и имеет права в Google Sheets / Drive. Развёртывание бота, переменные окружения и CLI — в [README.md](../README.md), детали снимка проекта — в [contract_project_snapshot.md](contract_project_snapshot.md). Текст для **магистранта** — [guide_user_ru.md](guide_user_ru.md), для **научного руководителя** — [guide_supervisor_ru.md](guide_supervisor_ru.md). Отправка справок и напоминаний в Telegram (личка, **`broadcast`**) — **§6**.

---

## 1. Как бот узнаёт администратора

- Права **не** привязаны к Google-аккаунту человека.
- В той же **Google-книге**, куда пишет бот (`SPREADSHEET_ID`), должен быть лист **`Администраторы`** с колонками (как минимум используется проверка по `telegram_id`):

```text
telegram_id, username, fio, role, active
```

- Ваш **числовой Telegram ID** должен быть в этом листе в колонке `telegram_id` (и запись должна считаться активной — см. реализацию в `magister_checking/bot/sheets_repo.py`).
- Узнать свой ID: через бота при `/start`, или сервисы вроде `@userinfobot`.
- Без записи в **`Администраторы`** команды `/admin`, `/project_card`, `/student_message`, **`/student_message_bulk`**, расширенный `/spravka`, `/stats`, `/sync_dashboard`, **`/sync_magistrants`** и превью научруковских списков **`/unreg` ФИО** / **`/reg_list` ФИО** будут отказаны.

## 2. Команды администратора (обзор)

| Команда | Назначение |
|---------|------------|
| `/start` | Те же сценарии регистрации / привязки, что и у магистранта (если вы тестируете анкету). В т.ч. при «да» после привязки по ФИО или после сводки анкеты возможен шаг **PIN** по телефону в таблице (см. **§7**, подраздел про PIN, и **§8.6**). |
| `/help` | Список команд **в расширении для администратора**. |
| `/admin` | Панель: **три** кнопки — PDF-карточка, **сообщение одному** магистранту и **групповое стандартное напоминание** по номерам строк (см. §6.2), либо переход по командам ниже. |
| `/project_card` | Запрос **номера строки** листа «Регистрация» или **ФИО** → полный пересчёт по строке, обновление листов, генерация **PDF-карточки** и отправка файла вам в чат. |
| `/student_message` | Сообщение **одному** магистранту в личку: после ввода строки или ФИО — шаблонное напоминание или свой текст (см. §6.2). |
| `/student_message_bulk` | То же **стандартное напоминание**, что в одном из шаблонов «одному», но для **нескольких строк** таблицы: список **номеров строк** без ФИО; у каждого своё имя из строки; см. §6.2. |
| `/spravka` | Меню: краткий текст, **полный текст для комиссии в чате**, **PDF** — с указанием **чужой** строки/ФИО; также приём **JSON-снимка** с Drive (см. раздел 4). |
| `/stats` | Краткая сводка по регистрациям — те же цифры, что на листе **`Dashboard`** (ячейка D1 и связанные данные), без открытия таблицы. |
| `/sync_dashboard` | Принудительно **пересчитать** лист **`Dashboard`** (актуально, если правили таблицу вручную вне бота). |
| `/sync_magistrants` | Пересчитать лист магистрантов (**`MAGISTRANTS_WORKSHEET_NAME`**, по умолчанию в `.env` не задан): колонка **«Регистрация»** (зарегистрирован / нет), нормализация **телефонов** к виду **+7…** и, **если в шапке листа есть колонка `telegram_id`**, подстановка **`telegram_id`** из «Регистрации» по паре ФИО + телефон. |
| `/unreg` **ФИО научрука** | **Только админ** (со словами после команды в той же строке). То же содержание, что научрук получает от **`/unreg`** без аргументов, но для **указанного** ФИО из листа **`научрук`**, например **`/unreg Иванов Иван Иванович`** или **`/unreg Иванов`**, если совпадение однозначно. Ответ помечен как превью. Научрук без прав администратора бот считает **своими** студентами по привязанному аккаунту — для него **`/unreg`** без аргументов. Требуются **`MAGISTRANTS_WORKSHEET_NAME`** и шапка «Магистранты». См. **§2.1** (админ без ФИО, админ+научрук). |
| `/reg_list` **ФИО научрука** | **Только админ** (формат как у строки выше): превью отчёта **`/reg_list`** для указанного научрука, например **`/reg_list Иванов Иван Иванович`**. См. **§2.1**. |
| `/recheck` | **Магистрант:** повторная проверка своей строки (по `telegram_id` в «Регистрации»). **Администратор:** можно указать **чужую** строку в той же команде — `/recheck 21` (номер строки листа «Регистрация») или `/recheck Иванов И.И.` (ФИО; при нескольких совпадениях бот попросит уточнить номер). Допустимы модификаторы **`quick`** / **`only-if-changed`** (прогон только если входы менялись с прошлого раза). Если вы админ и **нет своей строки** в таблице, без цели в команде бот попросит номер или ФИО **следующим сообщением** (или используйте сразу `/recheck N`). После ответа бота под сообщением есть кнопка **«Перепроверить»**: у администратора в callback подставляется **номер только что проверенной строки**, чтобы не вводить его снова. **Научный руководитель** «чужой» строки через **`/recheck`** не задаёт — тот же полный конвейер проверки по **подопечному** даёт **`/status ФИО`** (см. [guide_supervisor_ru.md](guide_supervisor_ru.md)); в «История проверок» для таких прогонов пишется отдельный **`source`** в коде. |
| `/cancel` | Выход из текущего диалога. |

Меню команд в Telegram подтягивается после перезапуска бота.

### 2.1. `/unreg` и `/reg_list`: админ против научрука

Команды **одни и те же**, смысл зависит от **роли** и **наличия аргументов** (всё в **личном** чате с ботом):

| Вы | Что отправить | Ответ бота |
|----|---------------|------------|
| **Только админ** (вашего `telegram_id` **нет** в листе **`научрук`** как у научрука) | **`/unreg ФИО`** или **`/reg_list ФИО`** — **ФИО научрука** в той же строке, после команды (можно несколько слов и **только фамилия**, если по данным книги человек один). | Превью того же текста, что у этого научрука по **`/unreg`** / **`/reg_list`** без аргументов (в начале сообщения будет пометка **превью**). |
| **Только админ**, без строки после команды | **`/unreg`** или **`/reg_list`** **без** ФИО. | Бот попросит **указать ФИО научрука** (пример в ответе). |
| **Админ и научрук** (вы в **`Администраторы`** и привязаны в **`научрук`**) | **`/unreg`** или **`/reg_list`** **без** аргументов. | Список **ваших** магистрантов с листа **«Магистранты»** — как у обычного научрука (не превью чужого). |
| **Админ и научрук** | **`/unreg ФИО`** / **`/reg_list ФИО`** **с** аргументами. | Превью для **указанного** научрука (имя из команды может отличаться от вашего ФИО в **`научрук`** — удобно куратору смотреть чужую группу). |

Научный руководитель **без** записи в **`Администраторы`**: допустим только вариант **без** аргументов (свой список); строка после команды к превью **других** не относится и для не‑админов не используется.

Предварительное условие для любых этих режимов: в **`.env`** задан **`MAGISTRANTS_WORKSHEET_NAME`** и на листе **«Магистранты»** корректная шапка (в т.ч. колонка научрука для отбора строк). Альтернатива без бота — подраздел **«Превью ответов научрука»** ниже (**§7**, CLI **`supervisor-list-preview`**).

## 3. PDF-карточка проекта: `/admin` и `/project_card`

1. Вызовите **`/admin`** и нажмите **«Сформировать карточку проекта»** (первая строка клавиатуры), либо воспользуйтесь сразу **`/project_card`**.
2. Введите **номер строки** (≥ 2) на листе **`Регистрация`** или **ФИО** магистранта (как в таблице).
3. Бот пересчитывает данные по отчёту и диссертации, **обновляет строку** в «Регистрации», обновляет **`Dashboard`**, формирует PDF и присылает файл.

Если строка не найдена или ФИО неоднозначно — бот вернёт сообщение об ошибке; при необходимости уточните номер строки вручную в таблице.

## 4. Справки и отчёты: `/spravka`

### 4.1. Режимы по чужой строке

После **`/spravka`** доступны (для админа):

- **Кратко (магистрант)** — полный прогон проверки для **указанной** строки/ФИО, затем короткий HTML-текст.
- **Полный текст (комиссия, чат)** — тот же прогон, вид **для комиссии** прямо в чате.
- **PDF для комиссии** — PDF по строке/ФИО.

Проверка может занять **до ~минуты**.

### 4.2. Снимок проекта из JSON (без повторного прогона)

Если у вас уже есть файл **`project_snapshot_*.json`** (или иной корректный **project snapshot** с Drive / из выгрузки бота):

1. В **личном чате** с ботом отправьте файл **документом** (расширение **`.json`**).
2. Бот распознаёт снимок, **не запускает** пайплайн заново и присылает **два** сообщения в HTML:

   - **«Как для магистранта»**;
   - **«Для комиссии»**.

Ограничение по размеру файла задаётся в коде (`_SNAPSHOT_JSON_MAX_BYTES` в `handlers.py`).

Формат снимка: **`schema_version`**, метаданные, фазы проверки, ссылки и т.д. — см. [contract_project_snapshot.md](contract_project_snapshot.md).

### 4.3. Запись JSON на Drive (опционально)

Если в `.env` настроены папки (**`PROJECT_CARD_OUTPUT_FOLDER_URL`** / **`PROJECT_SNAPSHOT_OUTPUT_FOLDER_URLS`** и права сервисного аккаунта **Content Manager** на Shared Drive), после проверки с записью строки могут сохраняться JSON-снимки — см. `.env.example` и комментарии в конфигурации (`magister_checking/bot/config.py`).

## 6. Справки магистрантам и напоминания (личка, Telegram)

Вся доставка **от бота** идёт **в личный чат** с ботом: для этого у магистранта в листе **«Регистрация»** должна быть заполнена колонка **`telegram_id`** (это сохраняется, когда человек хотя бы раз написал боту в личке и прошёл сценарий **`/start`**). Одно членство в **групповом чате** курса **не добавляет** ID в таблицу автоматически и **не является** источником адресатов для кода бота — см. подраздел **6.5**.

### 6.1. Отправить магистранту **полную справку** после проверки (как после `/recheck`)

Справка по этапам (HTML с блоками результатов и ссылок) может уйти студенту **без действий со стороны магистранта**, если:

1. Для нужной строки в **«Регистрации»** заполнен **`telegram_id`** (числовой ID из Telegram).
2. Вы выполнили запись результатов проверки в таблицу через CLI с флагом **`--notify-student`** (только вместе с **`--apply`**).

Пример:

```powershell
python -m magister_checking check-row --row 21 --apply --notify-student
```

Подробности параметров строки (**`--row`**, **`--fio`**) и ограничения — в [README.md](../README.md) и во флаге в `magister_checking/cli.py`. Если используете **`--only-if-changed`**, а прогон **сокращён** (вход не менялся), автоматическая отправка справки **не выполняется** — при необходимости уберите ограничение или попросите магистранта выполнить **`/recheck`** самостоятельно.

**Альтернатива:** магистрант сам вызывает **`/recheck`** в личке — тот же краткий вид справки приходит ему же.

### 6.2. Сообщения магистрантам **из бота** (`/admin`, `/student_message`, `/student_message_bulk`)

**Одному адресату — кнопка «Сообщение магистранту» или `/student_message`**

В личном чате с ботом: **`/admin`** → вторая строка клавиатуры **«Сообщение магистранту»** или команда **`/student_message`**.

1. Введите **номер строки** листа **«Регистрация»** или **ФИО** — так же, как для PDF-карточки (`/project_card`).
2. Выберите шаблон:
   - **стандартное напоминание** (призыв снова открыть бота, **`/start`** и **`/recheck`**);
   - **стандарт + до трёх строк** своих замечаний;
   - **только свой текст** одним сообщением.
3. После предпросмотра нажмите **«Отправить»** или **«Отмена»**. Если в конфигурации заданы папки (**`PROJECT_SNAPSHOT_OUTPUT_FOLDER_URLS`** / см. загрузку снимков) и для этой строки на Drive уже есть сохранённый JSON **`project_snapshot_r{номер}_…json`**, бот добавит **второе сообщение**: HTML-файл с полным текстом снимка проверки для комиссии (все документные ссылки **кликабельны** в браузере). Выбирается файл с **наибольшим временем изменения** на Drive среди папок. Если снимков нет — отправляется только текст напоминания (администратору приходит короткая подсказка).

**Группа магистрантов (`/student_message_bulk`):**

Из бота без Python: **`/admin`** → кнопка **«Групповое напоминание по строкам»** (третья строка клавиатуры) или команда **`/student_message_bulk`**. Указываются **только номера строк «Регистрации»** (через пробел, запятую или с новой строки). Каждому уходит **тот же стандартный текст**, что в пункте «стандартное напоминание» выше, с подстановкой **своего ФИО** из строки; при необходимости — второе сообщение с HTML-снимком с Drive (как у одиночной рассылки). Повтор одного и того же **`telegram_id`** в списке строк объединяется (одно сообщение на получателя). За один раз не больше **40** строк.

Сообщение уходит в **личку** магистранта по **`telegram_id`** из строки. Реализация: **`magister_checking/bot/handlers.py`** (`student_reminder_*`), загрузка снимка — **`magister_checking/drive_latest_snapshot.py`**.

### 6.3. Произвольный текст **одному или выбранным по строкам** магистрантам: `send-message`

Рассылка **с телефона без командной строки** — сценарии в **п. 6.2** (`/student_message`, `/student_message_bulk`). Ниже — **CLI** на сервере с компьютера.

Текст из файла (**UTF‑8**, без HTML-разметки) в **личку**:

- по **одной** строке листа — **`--row N`** (**`telegram_id`** из этой строки **«Регистрации»**);
- по **нескольким строкам** — **`--rows N …`** (номера через пробел, один текст всем указанным; см. ниже **`--rate`** для пауз между отправками);
- по **`--telegram-id`**, если ID известен.

Сначала **dry-run** (превью, без отправки), затем отправка с теми же двумя подтверждениями, что и у **`broadcast`**:

```powershell
python -m magister_checking send-message --message-file напоминание.txt --row 21
python -m magister_checking send-message --message-file напоминание.txt --row 21 --send --i-know-this-is-irreversible
python -m magister_checking send-message --message-file напоминание.txt --rows 5 7 9 10
python -m magister_checking send-message --message-file напоминание.txt --rows 5 7 9 10 --send --i-know-this-is-irreversible
```

Или: `--telegram-id 123456789` вместо **`--row`** / **`--rows`**. При **`--rows`** одинаковый **`telegram_id`** в двух строках даёт один адресат (дубликаты не отправляются дважды). Параметр **`--rate`** задаёт примерную частоту сообщений в секунду при нескольких адресатах (по умолчанию 25/c). Реализация: подкоманда **`send-message`** в **`magister_checking/cli.py`** (транспорт **`send_broadcast`**).

### 6.4. Массовое напоминание: `broadcast`

Если студент уже был в таблице (есть **`telegram_id`**), но **не заходит**, чтобы поправить данные:

- **Массовая рассылка** произвольного текста — CLI **`broadcast`** (всем из выбранного списка ID).

Порядок для **`broadcast`** (по умолчанию безопасный **dry-run**):

1. Подготовьте файл с текстом сообщения (**UTF‑8**), например `message.txt`.
2. Сначала **без** отправки посмотрите список адресатов:

   ```powershell
   python -m magister_checking broadcast --message-file message.txt
   ```

3. Осознанно отправьте (нужны **оба** флага — защита от случайной рассылки):

   ```powershell
   python -m magister_checking broadcast --message-file message.txt --send --i-know-this-is-irreversible
   ```

Источники адресатов (`--source`): **`registration`** — только **`telegram_id`** из листа **«Регистрация»**; **`persistence`** — ключи из файла состояния бота (**`BOT_PERSISTENCE_FILE`**; при отсутствии переменной — путь по умолчанию в `magister_checking/bot/config.py`, типичное имя `state/magistrcheckbot_state.pickle`); **`both`** (по умолчанию) — объединение с дедупликацией. Подробности — `magister_checking/broadcast.py` и **`cmd_broadcast`** в `magister_checking/cli.py`.

Если студент **заблокировал бота** или аккаунт удалён, Telegram вернёт **`Forbidden`** — такой адресат попадёт в отчёт об ошибках рассылки; понадобятся каналы вне Telegram (куратор, группа).

### 6.5. Сводка: что использовать когда

| Задача | Средство |
|--------|----------|
| Выслать **проверенную справку** после вашего **`check-row --apply`** | **`--notify-student`** (с непустым **`telegram_id`**) |
| Короткий повтор студентом своей строки | Магистрант: **`/recheck`** в личке |
| **Текстовое** напоминание **одному** из интерфейса бота (шаблоны) | **`/admin`** → **«Сообщение магистранту»** или **`/student_message`** |
| **Стандартное напоминание нескольким** из бота по списку номеров строк | **`/student_message_bulk`** или **«Групповое напоминание по строкам»** |
| **Текстовое** напоминание **одному или выбранным строкам** с сервера/скрипта | **`send-message`** (**`--row`**, **`--rows`**, **`--telegram-id`**) |
| **Общее** текстовое напоминание всем, у кого в таблице есть ID | **`broadcast`** |
| **Проверить в личке с бота** списки «как у научрука» (**незарегистриров.** / **зарегистриров.**) по чужому ФИО без входа под его Telegram | **`/unreg`** **ФИО научрука**, **`/reg_list`** **ФИО научрука** (см. **§2.1**); без Telegram — **`supervisor-list-preview`** (**§7**) |

### 6.6. Связано с документацией §8

Команда **`check-row`** и флаг **`--notify-student`** упомянуты также в **§8.3** вместе с ручным вводом ссылок.

### 6.7. Групповой чат и скриншот «кто вступил по ссылке»

Сообщения вида **«ФИО вступил(а) в группу по ссылке»** — это только **служба Telegram** по группам. Там видны **отображаемые имена**, а не надёжный ключ для связи со строкой **«Регистрации»**; смайлики и дубликаты имён затрудняют сопоставление.

**Как это использовать организационно (не кодом магистерского бота):**

- напоминание **на всю группу** — закреплённое сообщение, что нужно дописать данные через **личный** `/start`;
- для персональных напоминаний от **бота** опираться на **`telegram_id`** в таблице (личный диалог со **`/start`**), а скриншот группы — лишь **вспомогательный** человеческий контроль («кто ещё не заходил в личку»).

### Одноразовый PIN подтверждения телефона

Текст для магистранта в интерфейсе **`/help`** совпадает по смыслу с тем, что описано **[в руководстве пользователя](guide_user_ru.md)** (раздел 2):

**«При привязке к строке или перед сохранением анкеты бот может попросить одноразовый код по телефону из таблицы (см. лог бота в учебном контуре).»**

**Когда включается (после ответа «да», до записи `telegram_id` или финального сохранения анкеты):**

| Сценарий | Откуда берётся телефон для кода |
|----------|----------------------------------|
| Привязка к строке **«Регистрация»** по ФИО | Колонка **телефона** найденной строки (нормализация через `normalize_phone_ru_kz` в коде → вид **`+7` + 10 цифр**). |
| Новая регистрация («да» после **сводки** анкеты) | То, что пользователь ввёл в поле телефона, после такой же нормализации. |
| Привязка **администратора** или **научного руководителя** к строкам **`Администраторы`** / **`научрук`** | Ячейка телефона в этой строке (те же алиасы колонки, что для «Регистрации» — см. `magister_checking/bot/sheets_repo.py`). |

**Если после нормализации телефон пустой** (в таблице нет номера, пропуск поля в анкете, нераспознаваемый формат), бот **не выдаёт** PIN по этому шагу и завершает сохранение/привязку без него (`handlers.py`: в лог пишется краткая пометка, что код пропущен).

**Техника (MVP):** хеш кода (**SHA-256**), живые попытки **в памяти процесса** (`magister_checking/bot/pin_verify.py`): срок **~5 минут**, **до 3** неверных вводов, длина кода **4–6** цифр. После **`/cancel`** активная попытка снимается; после **перезапуска** процесса бота активные коды теряются.

**Поддержка:** подсказывать обратившемуся магистранту свериться с **консолью** или файлом **`BOT_LOG_FILE`** (см. **§8.6** ниже).

### Массовая нормализация телефонов в книге (CLI)

Стандарт в таблице — **`+7` и десять цифр** (как в `normalize_phone_ru_kz` / при **`/sync_magistrants`**). Одна команда с тем же сервисным аккаунтом, что у бота (`.env`):

```powershell
python -m magister_checking normalize-phones
```

Обновляет колонку **телефона** по алиасам заголовка на листах **`WORKSHEET_NAME`** (обычно «Регистрация») и **`научрук`**; если в `.env` задан **`MAGISTRANTS_WORKSHEET_NAME`**, дополнительно вызывается та же синхронизация, что и по **`/sync_magistrants`** (телефоны **+7**, колонка «Регистрация» и при наличии колонки — **`telegram_id`** на листе магистрантов). Лист **«Администраторы»** командой не трогается — при необходимости скопируйте подход через таблицу вручную или расширьте скрипт.

Реализация: **`normalize_phones_to_ru_kz_standard`** и **`normalize_worksheet_phones_ru_kz`** в `magister_checking/bot/sheets_repo.py`.

### Превью ответов научрука `/unreg` и `/reg_list` (бот и CLI)

Сводная таблица ролей и аргументов — в **§2.1** выше.

**В боте (предпочтительно в личке с бота как админ):** те же команды **`/unreg`** и **`/reg_list`**, но с **ФИО научрука** в той же строке после команды (как в листе **`научрук`**), например **`/unreg Иванов Иван Иванович`** и **`/reg_list Иванов`**. По сути это превью научруковских отчётов (в ответе — пометка «превью»). В **меню команд** пункты **`unreg`** и **`reg_list`** с краткой подсказкой для администратора; перезапуск бота нужен для обновления подписей.

**Подсказка по синтаксису Telegram:** пробел перед первой буквой ФИО — обязательна; несколько слов передаются **одним сообщением** (все слова после имени команды попадают в ФИО, например **`/reg_list Петрова Анна Сергеевна`**).

**Через консоль на сервере** — если удобнее сценарии без Telegram или автоматизация, используйте сервисный аккаунт из `.env`:

```powershell
python -m magister_checking supervisor-list-preview --telegram-id 123456789
```

или, если нужно обойти привязку по `telegram_id`, подставив эталонное ФИО из листа **`научрук`**:

```powershell
python -m magister_checking supervisor-list-preview --supervisor-fio "Амирова Гульнур Каргабековна"
```

- **`--kind unreg`** — только список незарегистрированных; **`--kind reg_list`** — только зарегистрированных; **`--kind both`** (по умолчанию) — оба блока подряд.
- Логика та же, что в **`magister_checking/bot/supervisor_lists.py`** (используется и ботом).

Сопоставление строк на листе **«Магистранты»** с научруком допускает **сокращённое имя и инициалы** при совпадении **фамилии** и согласованном **имени** (см. **`supervisor_name_matches`** в `sheets_repo.py`): например ячейка «Амирова Г. К.» и ФИО в **`научрук`** «Амирова Гульнур Каргабековна» считаются одним человеком. Поле «Научный руководитель» в **«Регистрации»** в эти списки не входит — магистрант привязывается по **ФИО + телефону**.

### 8.2. Что искать при проблемах с промежуточным отчётом и колонками L–O

**Не путать с «нет связи с Google».** Если в том же процессе в логе уже есть успех **Drive API** (например «snapshot saved» на Drive) или нормальная работа **Sheets**, значит сервисный аккаунт **аутентифицирован**, ключ к проекту **действителен**, обращение к Google **есть**. Сообщение про **Google Docs API** описывает **другой продукт API** в том же GCP-проекте: включение **Docs** и **Drive** задаётся **отдельными** переключателями в Console. Такую ошибку **неверно** пересказывать как отсутствие связи SaaS-сервис‑аккаунта с облаком вообще.

**Для искусственного интеллекта в Cursor** (агент читает его в каждой сессии): файл **`.cursor/rules/gcp-api-diagnostics.mdc`** — отдельно от этого руководства; в нём зафиксирована логика ответов по GCP, чтобы не путать Drive и Docs.

Строки пишет логгер **`magister_checking.bot.report_enrichment`** (обогащение строки регистрации из тела документа через **Google Docs API**).

**Какие форматы допускаются для промежуточного отчёта (колонка с URL отчёта):** помимо **Google Doc** по ссылке, на Drive разрешены нативные файлы **DOCX** и **PDF** (`application/vnd.openxmlformats-officedocument.wordprocessingml.document` и `application/pdf`). Для DOCX и PDF бот использует **Google Drive API** (`files.copy` с конвертацией в Google Doc), затем читает текст так же через **Google Docs API**, как для обычного Doc. Если в подпапке `drive_folder` ищется файл отчёта автоматически, приоритет по типам: Doc → DOCX → PDF.

Ищите в консоли или файле подстроки:

| Подстрока / паттерн | Смысл |
|---------------------|-------|
| `report_enrichment: documents().get(report) failed:` | Запрос к **`docs.googleapis.com/v1/documents/…`** к документу промежуточного отчёта **не выполнен**; до парсера таблицы код не дошёл; в Sheets тогда наш **служебный текст ошибки**, а не ссылки из отчёта. |
| `Google Docs API has not been used` / `SERVICE_DISABLED` | Бот жалуется, что сервис **Google Docs API** не включён в проекте GCP, к которому привязан **JSON ключ** (формулировку даёт ответ Google в теле ошибки). |
| `403` + `Forbidden` без `SERVICE_DISABLED` | Чаще **нет прав** сервисного аккаунта на сам документ отчёта — добавьте **email сервисного аккаунта** (поле **`client_email`** в JSON ключе) в доступ к файлу отчёта с правом читателя. |
| В логе есть полный **`HttpError`** | В нём текст Google и иногда **`activationUrl`** на включение API. |

Колонки «Проверка ссылки» / «Доступ открыт» для URL отчёта (валидность и HTTP как к **странице**) отрабатывают **отдельно** от чтения **тела документа** через Docs API; «yes» там **не означает** автоматический успех `documents().get`.

### 8.3. Ручной ввод четырёх ссылок (временный обход)

Если из текста промежуточного отчёта **не удаётся** заполнить колонки с **папкой магистерского проекта**, **заключением ЛКБ**, **диссертацией** и **публикацией** (сбой **Google Docs API**, разбор таблицы, доступ к файлу отчёта и т.п.), администратор может **вписать в лист «Регистрация» в эту строку** четыре обычных URL — в те же столбцы, куда бот пишет результаты проверки (ключи/подписи в заголовке: в духе ссылок на папку проекта, ЛКБ, диссертацию, публикацию; точное соответствие колонок задаётся заголовком книги и кодом `read_sheet_link_overrides_for_row` в `magister_checking/bot/sheets_repo.py`).

**Условия:**

- В ячейках должны быть **настоящие ссылки** (`https://…` / `http://…` на Google Doc, **DOCX или PDF** на Drive, папку — по правилам этапов 2–3), **не** длинные сообщения об ошибке из прошлых прогонов: такие строки бот **не считает** ссылками и не подставляет.
- У **сервисного аккаунта** (email из JSON ключа) должен быть **чтение** всех четырёх объектов и промежуточного отчёта, иначе этапы 3–4 снова упрётся в доступ или неподдерживаемый MIME (DOCX/PDF/Google Doc для отчёта — поддерживаются; другие типы файлов — нет).

**Действие:** после сохранения ячеек запустите для этой строки **повторную проверку** — **`/recheck`** у магистранта или **`python -m magister_checking check-row --row … --apply`** (см. [README.md](../README.md)). Прогон **объединяет** разбор отчёта (если он доступен) с введёнными в таблице URL: для «похожих на ссылку» ячеек **приоритет у листа**. Дальше выполняются проверки ссылок (этап 3), затем **страницы, число источников, соответствие оформлению** (те же правила, что и при обычной проверке: переменные **`FORMATTING_*`** в `.env`, см. `magister_checking/formatting_rules.py`), а при успешном чтении файла диссертации дописываются **название и язык диссертации** в соответствующие колонки. Если оформление вручную **не** проходить через `check-row`/`/recheck`, колонку «соответствие» нужно заполнять отдельно по тем же правилам (иначе числа страниц/источников и вердикт по ГОС/методичке расходятся). Смена любой из четырёх ссылок в листе меняет **fingerprint** режима «только если изменилось» (`--only-if-changed` / `quick`), чтобы не пропустить новый прогон.

Чтобы после **`check-row --apply`** магистрант **сразу получил в личку Telegram** ту же полную справку по этапам и полям, что и после **`/recheck`**, добавьте **`--notify-student`** (только вместе с **`--apply`**, в строке — непустой **`telegram_id`**). С **`--only-if-changed`**, если прогон **сокращён** (входы не менялись), уведомление **не** отправляется.

Это **временная** мера на случай недоступного разбора отчёта; восстановление **Docs API** и доступа к документу отчёта остаётся предпочтительным сценарием.

### 8.4. Лимит Google Sheets (429, «Read requests per minute»)

Иногда в консоли или в **`state/logs/bot.log`** появляется предупреждение вида **`Quota exceeded`** для **`sheets.googleapis.com`** и метрики **чтений в минуту на пользователя**. Это **не** означает «нет связи с Google»: ключ сервисного аккаунта действует, но за короткий интервал выполнилось **слишком много операций чтения** таблицы (полная проверка строки, синхронизация Dashboard, несколько **`/recheck`** подряд и т.п.).

**Что делать:** подождать **до минуты** и повторить команду; не запускать массу проверок одновременно из нескольких окон. Такие ошибки обрабатываются как временный лимит: в лог пишется **WARNING**, на **`BOT_ALERT_CHAT_IDS`** это **не дублируется** специально (чтобы не заспамить поддержку при кратковременной перегрузке квоты).

### 8.5. Алерты в Telegram

- При необработанной ошибке в хендлере бот может слать сообщения на **`BOT_ALERT_CHAT_IDS`** (список `chat_id` через запятую).

### 8.6. Одноразовый PIN в JSON-логе (`pin_issued` / `pin_verify`)

При включённом логировании процесса (консоль и/или **`BOT_LOG_FILE`**, см. §8.1) логгер **`magistrcheckbot`** пишет **JSON-объект в одну строку** (в файле) или текстовые строки (в консоли) с сообщениями кода верификации.

Ищите в файле записи **`"event": "pin_issued"`** и **`"event": "pin_verify"`**:

| Поле / момент | Смысл |
|---------------|--------|
| **`pin_plaintext`** (только при выдаче) | **Одноразовый код**, который в учебном контуре можно сверить со студентом по **ограниченному доступу к логу** (отдельной SMS в MVP нет). В продакшене храните лог с нужными правами; при SMS-провайдере текст кода можно перестать писать в лог и оставить только факт выдачи. |
| **`phone_normalized`** | Нормализованный ключ телефона (к чему привязана выдача). |
| **`status`** в `pin_verify` | В лог попадают значения вроде **`ok`**, **`wrong`**, **`expired`**, **`locked`** (исчерпаны попытки). |

Достаточно поиска по подстроке **`pin_issued`** или **`pin_verify`** в редакторе или `Select-String` в PowerShell. Реализация не считает неверный PIN «падением» сервера — ориентируйтесь по полю **`status`**.

Подробнее для студентов: [guide_user_ru.md](guide_user_ru.md) (PIN в разделе 2).

2. **Список включённых API (главное для скрина «что уже включено»):**  
   [APIs & Services → Enabled APIs и dashboard](https://console.cloud.google.com/apis/dashboard?project=magisterchecker)  
   Снимок всей страницы или несколько экранов, чтобы были видны **Google Sheets API**, **Google Drive API**, **Google Docs API** — все три нужны приложению (Sheets/Drive общий код; Docs — текст промежуточного отчёта).

3. **Библиотека (если хотите дополнительно один скрин на каждый API):**  
   [API Library](https://console.cloud.google.com/apis/library?project=magisterchecker) → поиск по имени API → страница API с кнопкой **Manage / Enable** уже в включённом состоянии.

4. **Service account (кто именно ходит в таблицы):**  
   [IAM → Service Accounts](https://console.cloud.google.com/iam-admin/serviceaccounts?project=magisterchecker)  
   Откройте аккаунт бота (**email** совпадает с **`client_email`** в ключе). Снимок экрана **Details** или списка с видимым **email**.

5. **Книга Google Sheets.** Откройте таблицу из **`SPREADSHEET_ID`** в браузере → **Поделиться** → скрин, где видно, что **`client_email`** сервисного аккаунта добавлен (**Редактор** для основной книги бота).

## 10. Формат справки магистранту после проверки

Краткий HTML после **`/recheck`** и у магистранта строится так: блок **«Результаты»** без дублирования длинных пояснений по ссылкам L/M/N/O — текст про тип ссылки (**папка вместо файла**, недоступность и т.д.) выводится **рядом с соответствующей строкой** в блоке **«Ссылки»** (в т.ч. у зачёркнутых URL). Подробное сравнение по оформлению диссертации (**«Найдено / Нужно»**) — в конце сообщения, блок **«Оформление (подробно)»**. Полный текст для комиссии по-прежнему в режимах **`/spravka`** — отдельный вид.
