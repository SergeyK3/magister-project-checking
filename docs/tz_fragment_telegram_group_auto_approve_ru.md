# Фрагмент ТЗ: оптимизация вступления в закрытую Telegram-группу (заявки + бот)

Документ предназначен для вставки в общее ТЗ. Описывает процесс работы с **супергруппой по заявкам**, **ботом с правами администратора** этой группы и **автоодобрением заявок** после проверки пользователя по данным Google Таблицы.

---

## 1. Цель

Обеспечить:

1. Вход в учебную Telegram-группу **только** для лиц, прошедших учётную логику проекта (магистранты, мастер-лист, администраторы, научные руководители).
2. **Минимум ручных** одобрений заявок: после выполнения условий заявка **одобряется ботом** через Telegram Bot API.
3. Корректную работу **как для новых** пользователей (сначала регистрация / привязка в боте), **так и для уже учтённых** («старых») — по совпадению **числового Telegram user id** с колонкой `telegram_id` в таблице.

---

## 2. Роли и инфраструктура Telegram

| Элемент | Требование |
|--------|------------|
| Группа | Супергруппа с режимом **вступления по заявке** (join request). |
| Бот | Добавлен в группу как **администратор** с правами, достаточными для **одобрения заявок на вступление** (в интерфейсе Telegram — право, связанное с приглашениями/заявками; конкретное имя права сверять с актуальной справкой Bot API и клиента). |
| Обновления | Бот получает от Telegram события о **новой заявке** (`chat_join_request` или эквивалент в используемой библиотеке). |
| Регистрация в боте | По-прежнему ведётся в **личном чате** с ботом; персональные данные анкеты **не** дублируются в групповой чат. |

---

## 3. «Белый список» для одобрения заявки

Пользователь считаётся **допущенным к автоодобрению** заявки в целевую группу, если **числовой идентификатор** его Telegram-аккаунта (`user_id` из заявки / из личного диалога) после нормализации **совпадает** с непустым значением в колонке **`telegram_id`** хотя бы в **одной** из следующих вкладок той же Google Таблицы, с которой работает бот:

1. **«Регистрация»** — учётная регистрация магистранта в процессе проекта.
2. **«Магистранты»** — мастер-лист магистрантов (колонка с Telegram ID в терминах заголовков таблицы, например `telegram_id` / согласованные алиасы).
3. **«Администраторы»** — служебный список администраторов процесса.
4. **«научрук»** — список научных руководителей.

### 3.1. Правила учёта строки на листе

- **Объединение источников (логика ИЛИ):** достаточно совпадения `telegram_id` **на любом одном** из перечисленных листов, при соблюдении правил строки для этого листа.
- **Нормализация идентификатора:** сравнение выполнять по строковому представлению после `strip` (и единого правила на отсутствие лишних символов), чтобы ячейки вида `123456` и числа совпадали.
- **Листы «Администраторы» и «научрук»:** строка учитывается только если пользователь **активен** — колонка **`active`** интерпретируется как логическое «да» / «yes» / «1» (как в текущей логике бота для проверки ролей). Пустая или отсутствующая колонка `active` — по решению реализации: либо трактовать как «неактивно», либо как «активно»; **в общем ТЗ зафиксировать один вариант** и не смешивать.
- **Лист «Регистрация»:** учитывать строку с непустым `telegram_id`, совпадающим с заявителем. При необходимости жёстче ограничить сценарий (например, только при завершённой анкете / заданном статусе заполнения) — **отдельное явное правило в ТЗ** и в коде одной точкой.
- **Лист «Магистранты»:** учитывать непустой `telegram_id` в сопоставленной колонке. При наличии колонок статуса («Регистрация» = зарегистрирован и т.п.) — по политике заказчика: либо любой непустой ID, либо только при статусе «зарегистрирован» / аналоге.

### 3.2. Целевая группа

- Идентификатор чата целевой супергруппы (`chat_id`) задаётся в конфигурации бота (например, переменная окружения). Автоодобрение выполняется **только** для заявок в этот чат; заявки в прочие чаты **не** одобряются автоматически.

---

## 4. Пользовательские сценарии

### 4.1. Уже учтённый пользователь («старый») — Telegram ID уже в таблице

1. Пользователь подаёт заявку на вступление в группу.
2. Бот получает событие заявки, извлекает `user_id`.
3. Выполняется проверка **«белого списка»** (п. 3): поиск совпадения `telegram_id` на листах **«Регистрация», «Магистранты», «Администраторы», «научрук»**.
4. При успешном совпадении заявка **сразу одобряется**; опционально — краткое сообщение в **личку** («Доступ к группе открыт»).
5. При отсутствии совпадения заявка **не** одобряется автоматически; в **личку** (если доставка возможна) — шаблон: «Вы не найдены в учётных списках по Telegram; завершите регистрацию у бота / обратитесь к администратору».

**Итог:** для старых пользователей достаточная проверка — **совпадение TG id с хранимым `telegram_id` на одном из четырёх листов** по правилам п. 3.1.

### 4.2. Новый пользователь — сначала заявка, потом регистрация в боте

1. Пользователь подаёт заявку → бот **сохраняет** ожидающую пару `(user_id, chat_id, время)` (устойчивое хранилище, переживает перезапуск).
2. В **личку** — инструкция открыть бота и пройти `/start` / регистрацию (и при необходимости привязку к строке «Регистрация» или сценарий научрука / админа на соответствующих листах).
3. После **успешного** завершения регистрации (запись в таблицу, появление актуального `telegram_id` на нужном листе) бот выполняет проверку «белого списка» и при наличии **необработанной** заявки для этого `user_id` в целевой группе вызывает **одобрение заявки**.
4. Уведомление в личку об успехе или о причине отказа (например, заявка уже истекла в Telegram).

### 4.3. Новый пользователь — сначала регистрация в боте, потом заявка

1. Пользователь проходит регистрацию в личке; его `telegram_id` попадает в таблицу (минимум на «Регистрация» или иной лист по сценарию).
2. Пользователь подаёт заявку в группу.
3. Бот по событию заявки проверяет «белый список» и при совпадении **сразу одобряет**.

### 4.4. Согласованность порядка действий

Требование: поддерживать **оба** порядка из п. 4.2 и 4.3 при условии доступности соответствующих событий Telegram API и хранения ожидающих заявок.

---

## 5. Функциональные требования (реализация)

- Реализовать **единую функцию** проверки: `user_id` → «находится ли в белом списке по четырём листам» с правилами п. 3.1.
- Реализовать **идемпотентное** одобрение: повторный вызов при уже принятом пользователе не должен ломать сценарий; ошибки API логировать, при необходимости уведомлять админские `chat_id` из существующей настройки алертов.
- Не одобрять заявки для чатов, не совпадающих с конфигурируемым **целевым** `chat_id`.
- Вести **аудит** в логах: событие заявки, результат проверки таблицы, факт одобрения или причина отказа.

---

## 6. Персональные и пограничные случаи

| Ситуация | Политика |
|----------|----------|
| В таблице пустой `telegram_id` (строка научрука без привязки) | Автоодобрение **невозможно**, пока пользователь не пройдёт привязку в боте или админ не заполнит ячейку. |
| Смена аккаунта Telegram | Совпадения с таблицей нет — только обновление `telegram_id` в таблице / повторная привязка. |
| Пользователь в «белом списке» по одному листу **и** имеет неактуальную строку на другом | Достаточно **одного** допустимого попадания (логика ИЛИ); устранение дубликатов — административная задача. |
| Бот потерял праска админа в группе | Одобрение невозможно; алерт администраторам, сообщение пользователю о ручном вмешательстве. |

---

## 7. Критерии приёмки (примерные)

1. Учётный пользователь с `telegram_id` **только** на «Регистрация» получает автоодобрение после заявки.
2. Учётный пользователь с `telegram_id` **только** на «Магистранты» — аналогично.
3. Администратор и научрук с `active=yes` и совпадающим `telegram_id` — автоодобрение.
4. Пользователь без совпадения ни на одном из четырёх листов — заявка **не** одобряется автоматически.
5. После регистрации нового магистранта ожидающая заявка (сценарий 4.2) одобряется без ручного действия в Telegram.
6. Без прав администратора у бота в группе — информативная ошибка в логах и отсутствие тихого «успеха».

---

## 8. Зависимости и актуализация

- Версия клиента Telegram Bot API / библиотеки — с поддержкой **join request** и **approve**.
- Заголовки и состав колонок листа «Магистранты» должны оставаться совместимыми с текущим распознаванием колонки Telegram ID в коде проекта (или явно обновляться в ТЗ и коде).
