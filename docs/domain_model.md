# Domain Model

## Основная идея

Система представляет собой workflow-платформу проверки магистерских проектов.

---

# Основные сущности

## User

Пользователь Telegram-бота.

### Атрибуты

* telegram_id
* role
* fio
* phone
* linked_row_id

---

## Role

Тип пользователя.

### Варианты

* magistrant
* supervisor
* admin

---

## Registration Row

Строка листа «Регистрация».

Основная сущность системы.

---

## Intermediate Report

Промежуточный отчет магистранта.

Источник ссылок для pipeline проверки.

---

## Dissertation

Документ диссертации.

Используется для:

* анализа страниц;
* анализа источников;
* проверки оформления;
* определения языка;
* извлечения названия.

---

## Project Snapshot

JSON-снимок состояния проекта.

### Содержит

* результаты проверок;
* ссылки;
* статусы этапов;
* extracted metadata;
* timestamps.

---

## Validation Pipeline

Последовательность этапов проверки.

### Этапы

1. Проверка промежуточного отчета
2. Проверка ссылок
3. Анализ диссертации
4. Извлечение метаданных
5. Формирование отчета

---

# Основные инварианты

## Registration Invariants

* один Telegram ID соответствует одному пользователю;
* пользователь имеет ровно одну роль;
* одна строка регистрации соответствует одному проекту.

---

## Pipeline Invariants

* pipeline идемпотентен;
* повторный запуск не должен создавать дубликаты;
* отсутствие части ссылок не должно ломать pipeline;
* ошибки API должны быть recoverable.

---

## FSM Invariants

* пользователь находится только в одном state;
* `/start` сбрасывает FSM;
* `/выход` очищает FSM;
* повторный `/справка` допустим в любой момент.

---

# Retry Model

Повторная проверка выполняется:

* повторным `/справка`;
* действиями администратора;
* внутренними retry-механизмами.

Отдельная команда `/recheck` не используется.
